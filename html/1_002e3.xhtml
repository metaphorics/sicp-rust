1.3 Formulating Abstractions with Higher-Order Procedures ¶

We have seen that procedures are, in effect, abstractions that describe compound operations on numbers independent of the particular numbers. For example, when we 

fn cube(x: i64) -> i64 {
    x * x * x
}

we are not talking about the cube of a particular number, but rather about a method for obtaining the cube of any number. Of course we could get along without ever defining this procedure, by always writing expressions such as 

3 * 3 * 3
x * x * x
y * y * y

and never mentioning cube explicitly. This would place us at a serious disadvantage, forcing us to work always at the level of the particular operations that happen to be primitives in the language (multiplication, in this case) rather than in terms of higher-level operations. Our programs would be able to compute cubes, but our language would lack the ability to express the concept of cubing. One of the things we should demand from a powerful programming language is the ability to build abstractions by assigning names to common patterns and then to work in terms of the abstractions directly. Procedures provide this ability. This is why all but the most primitive programming languages include mechanisms for defining procedures.

Yet even in numerical processing we will be severely limited in our ability to create abstractions if we are restricted to procedures whose parameters must be numbers. Often the same programming pattern will be used with a number of different procedures. To express such patterns as concepts, we will need to construct procedures that can accept procedures as arguments or return procedures as values. Procedures that manipulate procedures are called higher-order procedures. This section shows how higher-order procedures can serve as powerful abstraction mechanisms, vastly increasing the expressive power of our language.

1.3.1 Procedures as Arguments ¶

Consider the following three procedures. The first computes the sum of the integers from a through b: 

fn sum_integers(a: i64, b: i64) -> i64 {
    if a > b {
        0
    } else {
        a + sum_integers(a + 1, b)
    }
}

The second computes the sum of the cubes of the integers in the given range: 

fn sum_cubes(a: i64, b: i64) -> i64 {
    if a > b {
        0
    } else {
        cube(a) + sum_cubes(a + 1, b)
    }
}

The third computes the sum of a sequence of terms in the series \[ % :17: \frac{1}{1\cdot 3} + \frac{1}{5\cdot 7} + \frac{1}{9\cdot 11} + {\dots,} \] which converges to {\pi / 8} (very slowly):44 

fn pi_sum(a: i64, b: i64) -> f64 {
    if a > b {
        0.0
    } else {
        1.0 / (a as f64 * (a as f64 + 2.0)) + pi_sum(a + 4, b)
    }
}

These three procedures clearly share a common underlying pattern. They are for the most part identical, differing only in the name of the procedure, the function of a used to compute the term to be added, and the function that provides the next value of a. We could generate each of the procedures by filling in slots in the same template: 

fn ⟨name⟩(a: i64, b: i64) -> i64 {
    if a > b {
        0
    } else {
        ⟨term⟩(a) + ⟨name⟩(⟨next⟩(a), b)
    }
}

The presence of such a common pattern is strong evidence that there is a useful abstraction waiting to be brought to the surface. Indeed, mathematicians long ago identified the abstraction of summation of a series and invented “sigma notation,” for example \[ % :18: {\sum_{n = a}^b f(n)} \,=\, {f(a)} + \dots + {f(b),} \] to express this concept. The power of sigma notation is that it allows mathematicians to deal with the concept of summation itself rather than only with particular sums—for example, to formulate general results about sums that are independent of the particular series being summed.

Similarly, as program designers, we would like our language to be powerful enough so that we can write a procedure that expresses the concept of summation itself rather than only procedures that compute particular sums. We can do so readily in our procedural language by taking the common template shown above and transforming the “slots” into formal parameters: 

fn sum<F, N>(term: F, a: i64, next: N, b: i64) -> i64
where
    F: Fn(i64) -> i64,
    N: Fn(i64) -> i64,
{
    if a > b {
        0
    } else {
        term(a) + sum(term, next(a), next, b)
    }
}

Notice that sum takes as its arguments the lower and upper bounds a and b together with the procedures term and next. In Rust, we use generic type parameters F and N with trait bounds Fn(i64) -> i64 to specify that these parameters are functions that take an i64 and return an i64. The Fn trait represents functions and closures that can be called multiple times without consuming themselves. We can use sum just as we would any procedure. For example, we can use it (along with a procedure inc that increments its argument by 1) to define sum_cubes: 

fn inc(n: i64) -> i64 {
    n + 1
}

fn sum_cubes(a: i64, b: i64) -> i64 {
    sum(cube, a, inc, b)
}

Using this, we can compute the sum of the cubes of the integers from 1 to 10: 

sum_cubes(1, 10)
// => 3025

With the aid of an identity procedure to compute the term, we can define sum_integers in terms of sum: 

fn identity(x: i64) -> i64 {
    x
}

fn sum_integers(a: i64, b: i64) -> i64 {
    sum(identity, a, inc, b)
}

Then we can add up the integers from 1 to 10: 

sum_integers(1, 10)
// => 55

We can also define pi_sum in the same way:45 

fn pi_sum(a: i64, b: i64) -> f64 {
    fn pi_term(x: i64) -> f64 {
        1.0 / (x as f64 * (x as f64 + 2.0))
    }
    fn pi_next(x: i64) -> i64 {
        x + 4
    }
    sum(pi_term, a, pi_next, b) as f64
}

Using these procedures, we can compute an approximation to \pi: 

8.0 * pi_sum(1, 1000)
// => 3.139592655589783

Once we have sum, we can use it as a building block in formulating further concepts. For instance, the definite integral of a function f between the limits a and b can be approximated numerically using the formula \[ % :19: {\int_a^b \kern-0.3em f} \;=\; {\left[\;f\left(a + \frac{d x}{2}\right)\right.} \,+\, {f\left(a + dx + \frac{dx}{2}\right)} \,+\, {\left. f\left(a + 2dx + \frac{dx}{2}\right) \,+\, \dots \;\right] dx} \] for small values of {dx}. We can express this directly as a procedure: 

fn integral<F>(f: F, a: f64, b: f64, dx: f64) -> f64
where
    F: Fn(f64) -> f64,
{
    fn sum_f64<F, N>(term: F, a: f64, next: N, b: f64) -> f64
    where
        F: Fn(f64) -> f64,
        N: Fn(f64) -> f64,
    {
        if a > b {
            0.0
        } else {
            term(a) + sum_f64(term, next(a), next, b)
        }
    }

    let add_dx = |x| x + dx;
    sum_f64(f, a + dx / 2.0, add_dx, b) * dx
}

integral(|x| x * x * x, 0.0, 1.0, 0.01)
// => 0.24998750000000042

integral(|x| x * x * x, 0.0, 1.0, 0.001)
// => 0.249999875000001

(The exact value of the integral of cube between 0 and 1 is 1/4.)

Note that in the integral function, we use Rust’s closure syntax to define add_dx: the expression |x| x + dx creates a closure that captures the dx variable from the enclosing scope. The vertical bars |...| enclose the parameter list, and the expression after them is the closure body. We also pass a closure |x| x * x * x directly as an argument to integral, demonstrating that closures can be created inline without naming them.

1.3.2 Constructing Procedures Using Closures ¶

In using sum as in Procedures as Arguments, it seems terribly awkward to have to define trivial procedures such as pi_term and pi_next just so we can use them as arguments to our higher-order procedure. Rather than define pi_next and pi_term, it would be more convenient to have a way to directly specify “the procedure that returns its input incremented by 4” and “the procedure that returns the reciprocal of its input times its input plus 2.” We can do this by using Rust\’s closure syntax, which creates anonymous functions. Using closures we can describe what we want as 

|x| x + 4

and 

|x| 1.0 / (x * (x + 2.0))

Then our pi_sum procedure can be expressed without defining any auxiliary procedures as 

fn pi_sum(a: i32, b: i32) -> f64 {
    sum(|x| 1.0 / (x as f64 * (x as f64 + 2.0)),
        a,
        |x| x + 4,
        b)
}

Again using closures, we can write the integral procedure without having to define the auxiliary procedure add_dx: 

fn integral<F>(f: F, a: f64, b: f64, dx: f64) -> f64
where
    F: Fn(f64) -> f64,
{
    sum(f, a + dx / 2.0, |x| x + dx, b) * dx
}

In general, closures are used to create anonymous functions in the same way as fn, except that no name is specified for the function: 

|⟨formal-parameters⟩| ⟨body⟩

The resulting closure is just as much a function as one that is created using fn. The only difference is that it has not been associated with any name in the environment. In fact, 

fn plus4(x: i32) -> i32 { x + 4 }

is equivalent to 

let plus4 = |x: i32| x + 4;

We can read a closure expression as follows: 

|       x       |       x   +     4
|       |       |       |   |     |
the function of an argument x that adds x and 4

Like any expression that has a function as its value, a closure expression can be used immediately such as 

(|x, y, z| x + y + square(z))(1, 2, 3)
// => 12

or, more generally, in any context where we would normally use a function name.48

Using let to create local variables ¶

In Rust, the let keyword is used to create local variables. We often need local variables in our procedures other than those that have been bound as formal parameters. For example, suppose we wish to compute the function \[ % :22: {f(x,y)} \,=\, {x(1 + xy)^2} + {y(1 - y)} + {(1 + xy)(1 - y),} \] which we could also express as \[ % :23: \begin{eqnarray} a &=& {1 + xy,} \\ \hphantom{(x,y)} b &=& {1 - y,} \\ {f(x,y)} &=& {xa^2} + {yb} + {ab.} \end{eqnarray} \] In writing a procedure to compute f, we would like to include as local variables not only x and y but also the names of intermediate quantities like a and b. One way to accomplish this is to use an auxiliary procedure to bind the local variables: 

fn f(x: f64, y: f64) -> f64 {
    fn f_helper(x: f64, y: f64, a: f64, b: f64) -> f64 {
        x * square(a) + y * b + a * b
    }
    f_helper(x, y, 1.0 + x * y, 1.0 - y)
}

Of course, we could use a closure to specify an anonymous function for binding our local variables. The body of f then becomes a single call to that closure: 

fn f(x: f64, y: f64) -> f64 {
    (|a, b| x * square(a) + y * b + a * b)(1.0 + x * y, 1.0 - y)
}

In Rust, let bindings are a natural and convenient way to create local variables. Using let, the f procedure could be written as 

fn f(x: f64, y: f64) -> f64 {
    let a = 1.0 + x * y;
    let b = 1.0 - y;
    x * square(a) + y * b + a * b
}

The general form of a let binding in Rust is 

let ⟨var₁⟩ = ⟨exp₁⟩;
let ⟨var₂⟩ = ⟨exp₂⟩;
…
let ⟨varₙ⟩ = ⟨expₙ⟩;
⟨body⟩

which can be thought of as saying 

let ⟨var₁⟩ have the value ⟨exp₁⟩ and
    ⟨var₂⟩ have the value ⟨exp₂⟩ and
    …
    ⟨varₙ⟩ have the value ⟨expₙ⟩
  in ⟨body⟩

Each let binding associates a name with the value of the corresponding expression. The body of the function is evaluated with these names bound as local variables. In Rust, let bindings are evaluated sequentially, which means each binding can use the values of previous bindings. This is similar to using immediately-invoked closures: 

(|⟨var₁⟩, …, ⟨varₙ⟩| ⟨body⟩)(⟨exp₁⟩, …, ⟨expₙ⟩)

While Rust\’s let is a fundamental language construct rather than syntactic sugar, the conceptual relationship to closures helps understand how local variables create scoped bindings.

The scope of a variable specified by a let binding is from the point of binding to the end of the enclosing block. This implies that:

  • Let allows one to bind variables as locally as possible to where they are to be used. For example, if the value of x is 5, the value of the expression 
    {
    let inner_result = {
        let x = 3;
        x + x * 10
    };
    inner_result + x
}

    is 38. Here, the x in the inner block is 3, so the value of the inner expression is 33. On the other hand, the x that is the second argument to the outermost + is still 5.

  • In Rust, let bindings are evaluated sequentially. This matters when the expressions that provide the values for the local variables depend upon variables having the same names as the local variables themselves. For example, if the value of x is 2, the expression 
    {
    let x = 3;
    let y = x + 2;
    x * y
}

    will have the value 15 because y uses the new value of x (which is 3), so y will be 5, and x * y = 15. This is different from Scheme\’s let, where all bindings are evaluated before any are bound.

In Rust, let bindings are the natural way to define local variables within a function. For example, the procedure f above was written as 

fn f(x: f64, y: f64) -> f64 {
    let a = 1.0 + x * y;
    let b = 1.0 - y;
    x * square(a) + y * b + a * b
}

We use let bindings for local variables and values, while we use internal fn definitions only for internal helper functions.49

1.3.3 Procedures as General Methods ¶

We introduced compound procedures in Compound Procedures as a mechanism for abstracting patterns of numerical operations so as to make them independent of the particular numbers involved. With higher-order procedures, such as the integral procedure of Procedures as Arguments, we began to see a more powerful kind of abstraction: procedures used to express general methods of computation, independent of the particular functions involved. In this section we discuss two more elaborate examples—general methods for finding zeros and fixed points of functions—and show how these methods can be expressed directly as procedures.

Finding roots of equations by the half-interval method ¶

The half-interval method is a simple but powerful technique for finding roots of an equation {f(x) = 0}, where f is a continuous function. The idea is that, if we are given points a and b such that {f(a) < 0 < f(b)}, then f must have at least one zero between a and b. To locate a zero, let x be the average of a and b, and compute {f(x)}. If {f(x) > 0}, then f must have a zero between a and x. If {f(x) < 0}, then f must have a zero between x and b. Continuing in this way, we can identify smaller and smaller intervals on which f must have a zero. When we reach a point where the interval is small enough, the process stops. Since the interval of uncertainty is reduced by half at each step of the process, the number of steps required grows as {\Theta(\log(L\, /\, T))}, where L is the length of the original interval and T is the error tolerance (that is, the size of the interval we will consider “small enough”). Here is a procedure that implements this strategy: 

fn search<F>(f: F, neg_point: f64, pos_point: f64) -> f64
where
    F: Fn(f64) -> f64,
{
    let midpoint = average(neg_point, pos_point);
    if close_enough(neg_point, pos_point) {
        midpoint
    } else {
        let test_value = f(midpoint);
        if test_value > 0.0 {
            search(f, neg_point, midpoint)
        } else if test_value < 0.0 {
            search(f, midpoint, pos_point)
        } else {
            midpoint
        }
    }
}

We assume that we are initially given the function f together with points at which its values are negative and positive. We first compute the midpoint of the two given points. Next we check to see if the given interval is small enough, and if so we simply return the midpoint as our answer. Otherwise, we compute as a test value the value of f at the midpoint. If the test value is positive, then we continue the process with a new interval running from the original negative point to the midpoint. If the test value is negative, we continue with the interval from the midpoint to the positive point. Finally, there is the possibility that the test value is 0, in which case the midpoint is itself the root we are searching for.

To test whether the endpoints are “close enough” we can use a procedure similar to the one used in Example: Square Roots by Newton’s Method for computing square roots:50 

fn close_enough(x: f64, y: f64) -> bool {
    (x - y).abs() < 0.001
}

Search is awkward to use directly, because we can accidentally give it points at which f’s values do not have the required sign, in which case we get a wrong answer. Instead we will use search via the following procedure, which checks to see which of the endpoints has a negative function value and which has a positive value, and calls the search procedure accordingly. If the function has the same sign on the two given points, the half-interval method cannot be used, in which case the procedure signals an error.51 

fn half_interval_method<F>(f: F, a: f64, b: f64) -> f64
where
    F: Fn(f64) -> f64,
{
    let a_value = f(a);
    let b_value = f(b);
    if a_value < 0.0 && b_value > 0.0 {
        search(f, a, b)
    } else if b_value < 0.0 && a_value > 0.0 {
        search(f, b, a)
    } else {
        panic!("Values are not of opposite sign {} {}", a, b)
    }
}

The following example uses the half-interval method to approximate \pi as the root between 2 and 4 of {\sin x = 0}: 

half_interval_method(f64::sin, 2.0, 4.0)
// => 3.14111328125

Here is another example, using the half-interval method to search for a root of the equation {x^3 - 2x - 3 = 0} between 1 and 2: 

half_interval_method(
    |x| x * x * x - 2.0 * x - 3.0,
    1.0,
    2.0
)
// => 1.89306640625

Finding fixed points of functions ¶

A number x is called a fixed point of a function f if x satisfies the equation {f(x) = x}. For some functions f we can locate a fixed point by beginning with an initial guess and applying f repeatedly, \[ % :24: {f(x),}\quad {f(f(x)),}\quad {f(f(f(x))),} \quad{\dots,} \] until the value does not change very much. Using this idea, we can devise a procedure fixed_point that takes as inputs a function and an initial guess and produces an approximation to a fixed point of the function. We apply the function repeatedly until we find two successive values whose difference is less than some prescribed tolerance: 

const TOLERANCE: f64 = 0.00001;

fn fixed_point<F>(f: F, first_guess: f64) -> f64
where
    F: Fn(f64) -> f64,
{
    fn close_enough(v1: f64, v2: f64) -> bool {
        (v1 - v2).abs() < TOLERANCE
    }
    fn try_guess<F>(f: &F, guess: f64) -> f64
    where
        F: Fn(f64) -> f64,
    {
        let next = f(guess);
        if close_enough(guess, next) {
            next
        } else {
            try_guess(f, next)
        }
    }
    try_guess(&f, first_guess)
}

For example, we can use this method to approximate the fixed point of the cosine function, starting with 1 as an initial approximation:52 

fixed_point(f64::cos, 1.0)
// => 0.7390822985224023

Similarly, we can find a solution to the equation {y = \sin y + \cos y}: 

fixed_point(|y| y.sin() + y.cos(), 1.0)
// => 1.2587315962971173

The fixed-point process is reminiscent of the process we used for finding square roots in Example: Square Roots by Newton’s Method. Both are based on the idea of repeatedly improving a guess until the result satisfies some criterion. In fact, we can readily formulate the square-root computation as a fixed-point search. Computing the square root of some number x requires finding a y such that {y^2 = x}. Putting this equation into the equivalent form {y = x / y}, we recognize that we are looking for a fixed point of the function53 {y \mapsto x / y}, and we can therefore try to compute square roots as 

fn sqrt(x: f64) -> f64 {
    fixed_point(|y| x / y, 1.0)
}

Unfortunately, this fixed-point search does not converge. Consider an initial guess y_1. The next guess is {y_2 = x / y_1} and the next guess is y_3 = {x / y_2} = {x / (x / y_1)} = y_1. This results in an infinite loop in which the two guesses y_1 and y_2 repeat over and over, oscillating about the answer.

One way to control such oscillations is to prevent the guesses from changing so much. Since the answer is always between our guess y and {x / y}, we can make a new guess that is not as far from y as {x / y} by averaging y with {x / y}, so that the next guess after y is {{1\over2}(y + x / y)} instead of {x / y}. The process of making such a sequence of guesses is simply the process of looking for a fixed point of y \mapsto {{1\over2}(y + x / y)}: 

fn sqrt(x: f64) -> f64 {
    fixed_point(|y| average(y, x / y), 1.0)
}

(Note that y = {{1\over2}(y + x / y)} is a simple transformation of the equation {y = x / y;} to derive it, add y to both sides of the equation and divide by 2.)

With this modification, the square-root procedure works. In fact, if we unravel the definitions, we can see that the sequence of approximations to the square root generated here is precisely the same as the one generated by our original square-root procedure of Example: Square Roots by Newton’s Method. This approach of averaging successive approximations to a solution, a technique that we call average damping, often aids the convergence of fixed-point searches.

1.3.4 Procedures as Returned Values ¶

The above examples demonstrate how the ability to pass procedures as arguments significantly enhances the expressive power of our programming language. We can achieve even more expressive power by creating procedures whose returned values are themselves procedures.

We can illustrate this idea by looking again at the fixed-point example described at the end of Procedures as General Methods. We formulated a new version of the square-root procedure as a fixed-point search, starting with the observation that \sqrt{x} is a fixed-point of the function {y \mapsto x / y}. Then we used average damping to make the approximations converge. Average damping is a useful general technique in itself. Namely, given a function f, we consider the function whose value at x is equal to the average of x and {f(x)}.

We can express the idea of average damping by means of the following procedure: 

fn average_damp<F>(f: F) -> impl Fn(f64) -> f64
where
    F: Fn(f64) -> f64 + 'static,
{
    move |x| average(x, f(x))
}

In Rust, returning a closure requires using impl Fn as the return type. The move keyword captures f by value, transferring ownership into the returned closure. This is necessary because the closure outlives the function scope. The 'static lifetime bound ensures f doesn’t contain references with limited lifetimes.

Average-damp is a procedure that takes as its argument a procedure f and returns as its value a procedure (produced by the closure) that, when applied to a number x, produces the average of x and f(x). For example, applying average_damp to the square procedure produces a procedure whose value at some number x is the average of x and x^2. Applying this resulting procedure to 10 returns the average of 10 and 100, or 55:54 

average_damp(square)(10.0)
// => 55.0

Using average_damp, we can reformulate the square-root procedure as follows: 

fn sqrt(x: f64) -> f64 {
    fixed_point(
        average_damp(move |y| x / y),
        1.0,
    )
}

Notice how this formulation makes explicit the three ideas in the method: fixed-point search, average damping, and the function {y \mapsto x / y}. It is instructive to compare this formulation of the square-root method with the original version given in Example: Square Roots by Newton’s Method. Bear in mind that these procedures express the same process, and notice how much clearer the idea becomes when we express the process in terms of these abstractions. In general, there are many ways to formulate a process as a procedure. Experienced programmers know how to choose procedural formulations that are particularly perspicuous, and where useful elements of the process are exposed as separate entities that can be reused in other applications. As a simple example of reuse, notice that the cube root of x is a fixed point of the function {y \mapsto x / y^2}, so we can immediately generalize our square-root procedure to one that extracts cube roots:55 

fn cube_root(x: f64) -> f64 {
    fixed_point(
        average_damp(move |y| x / square(y)),
        1.0,
    )
}

Newton’s method ¶

When we first introduced the square-root procedure, in Example: Square Roots by Newton’s Method, we mentioned that this was a special case of Newton’s method. If {x \mapsto g(x)} is a differentiable function, then a solution of the equation {g(x) = 0} is a fixed point of the function {x \mapsto f(x)} where \[ % :28: {f(x)} \,=\, x - \frac{g(x)}{D g(x)} \] and {Dg(x)} is the derivative of g evaluated at x. Newton’s method is the use of the fixed-point method we saw above to approximate a solution of the equation by finding a fixed point of the function {f}.56

For many functions g and for sufficiently good initial guesses for x, Newton’s method converges very rapidly to a solution of {g(x) = 0}.57

In order to implement Newton’s method as a procedure, we must first express the idea of derivative. Note that “derivative,” like average damping, is something that transforms a function into another function. For instance, the derivative of the function {x \mapsto x^3} is the function {x \mapsto 3x^2}. In general, if g is a function and {dx} is a small number, then the derivative {Dg} of g is the function whose value at any number x is given (in the limit of small {dx}) by \[ % :29: Dg(x) \,=\, {\frac{g(x + dx) - g(x)}{dx}.} \] Thus, we can express the idea of derivative (taking {dx} to be, say, 0.00001) as the procedure 

const DX: f64 = 0.00001;

fn deriv<G>(g: G) -> impl Fn(f64) -> f64
where
    G: Fn(f64) -> f64 + 'static,
{
    move |x| (g(x + DX) - g(x)) / DX
}

Like average_damp, deriv is a procedure that takes a procedure as argument and returns a procedure as value. For example, to approximate the derivative of {x \mapsto x^3} at 5 (whose exact value is 75) we can evaluate 

fn cube(x: f64) -> f64 {
    x * x * x
}

deriv(cube)(5.0)
// => 75.00014999664018

With the aid of deriv, we can express Newton’s method as a fixed-point process: 

fn newton_transform<G>(g: G) -> impl Fn(f64) -> f64
where
    G: Fn(f64) -> f64 + Clone + 'static,
{
    let g_clone = g.clone();
    move |x| x - (g(x) / deriv(g_clone.clone())(x))
}

fn newtons_method<G>(g: G, guess: f64) -> f64
where
    G: Fn(f64) -> f64 + Clone + 'static,
{
    fixed_point(newton_transform(g), guess)
}

Here, newton_transform requires G: Clone because we need to use g both directly and pass it to deriv. We clone g to satisfy Rust’s ownership rules.

The newton_transform procedure expresses the formula at the beginning of this section, and newtons_method is readily defined in terms of this. It takes as arguments a procedure that computes the function for which we want to find a zero, together with an initial guess. For instance, to find the square root of x, we can use Newton’s method to find a zero of the function {y \mapsto y^2 - x} starting with an initial guess of 1.58

This provides yet another form of the square-root procedure: 

fn sqrt(x: f64) -> f64 {
    newtons_method(move |y| square(y) - x, 1.0)
}

Abstractions and first-class procedures ¶

We’ve seen two ways to express the square-root computation as an instance of a more general method, once as a fixed-point search and once using Newton’s method. Since Newton’s method was itself expressed as a fixed-point process, we actually saw two ways to compute square roots as fixed points. Each method begins with a function and finds a fixed point of some transformation of the function. We can express this general idea itself as a procedure: 

fn fixed_point_of_transform<G, T, F>(g: G, transform: T, guess: f64) -> f64
where
    G: Fn(f64) -> f64 + 'static,
    T: Fn(G) -> F,
    F: Fn(f64) -> f64,
{
    fixed_point(transform(g), guess)
}

This function takes a function g, a transform that converts one function into another, and an initial guess. The generic parameters G, T, and F allow flexible composition of different function transformations.

This very general procedure takes as its arguments a procedure g that computes some function, a procedure that transforms g, and an initial guess. The returned result is a fixed point of the transformed function.

Using this abstraction, we can recast the first square-root computation from this section (where we look for a fixed point of the average-damped version of {y \mapsto x / y}) as an instance of this general method: 

fn sqrt(x: f64) -> f64 {
    fixed_point_of_transform(move |y| x / y, average_damp, 1.0)
}

Similarly, we can express the second square-root computation from this section (an instance of Newton’s method that finds a fixed point of the Newton transform of {y \mapsto y^2 - x}) as 

fn sqrt(x: f64) -> f64 {
    fixed_point_of_transform(move |y| square(y) - x, newton_transform, 1.0)
}

We began section Formulating Abstractions with Higher-Order Procedures with the observation that compound procedures are a crucial abstraction mechanism, because they permit us to express general methods of computing as explicit elements in our programming language. Now we’ve seen how higher-order procedures permit us to manipulate these general methods to create further abstractions.

As programmers, we should be alert to opportunities to identify the underlying abstractions in our programs and to build upon them and generalize them to create more powerful abstractions. This is not to say that one should always write programs in the most abstract way possible; expert programmers know how to choose the level of abstraction appropriate to their task. But it is important to be able to think in terms of these abstractions, so that we can be ready to apply them in new contexts. The significance of higher-order procedures is that they enable us to represent these abstractions explicitly as elements in our programming language, so that they can be handled just like other computational elements.

In general, programming languages impose restrictions on the ways in which computational elements can be manipulated. Elements with the fewest restrictions are said to have first-class status. Some of the “rights and privileges” of first-class elements are:59

  • They may be named by variables.
  • They may be passed as arguments to procedures.
  • They may be returned as the results of procedures.
  • They may be included in data structures.60

Lisp, unlike other common programming languages, awards procedures full first-class status. This poses challenges for efficient implementation, but the resulting gain in expressive power is enormous.61

1.3.5 Iterator Combinators ¶

Throughout this chapter, we’ve explored procedures that operate on other procedures—higher-order procedures that abstract common patterns of computation. We’ve seen sum, product, and accumulate as ways to capture the essence of iterative combination. Rust takes this idea further with iterators—a zero-cost abstraction for sequential data processing that combines the elegance of higher-order functions with the performance of hand-written loops.

The Iterator Trait ¶

An iterator is any type that implements the Iterator trait, which requires a single method: 

pub trait Iterator {
    type Item;

    fn next(&mut self) -> Option<Self::Item>;
}

The next method returns Some(value) for each element in the sequence, and None when the sequence is exhausted. This simple interface enables powerful compositional patterns.

Consider counting from 1 to 5. We can create an iterator from a range: 

let range = 1..=5;
let mut iter = range.into_iter();

assert_eq!(iter.next(), Some(1));
assert_eq!(iter.next(), Some(2));
assert_eq!(iter.next(), Some(3));
assert_eq!(iter.next(), Some(4));
assert_eq!(iter.next(), Some(5));
assert_eq!(iter.next(), None);

The into_iter method comes from the IntoIterator trait, which types implement to convert themselves into iterators. Collections like arrays, vectors, and ranges all implement IntoIterator, allowing them to be used in for loops and with iterator methods.

Map, Filter, and Higher-Order Operations ¶

Recall the sum procedure from Procedures as Arguments, which abstracted the pattern of accumulating values across a range. With iterators, we can separate the concerns of generating values, transforming values, and combining values: 

// Sum of squares from 1 to 10
let sum_of_squares: i32 = (1..=10)
    .map(|x| x * x)
    .sum();

assert_eq!(sum_of_squares, 385);

Here, map is an iterator combinator—a method that transforms one iterator into another. It takes a closure and produces a new iterator that applies the closure to each element. The sum method is a consumer—it consumes the iterator and produces a final value.

Compare this to our earlier sum_integers from Procedures as Arguments: 

fn sum<F>(term: F, a: i32, next: fn(i32) -> i32, b: i32) -> i32
where
    F: Fn(i32) -> i32,
{
    if a > b {
        0
    } else {
        term(a) + sum(term, next(a), next, b)
    }
}

fn sum_cubes(a: i32, b: i32) -> i32 {
    sum(|x| x * x * x, a, |x| x + 1, b)
}

With iterators, this becomes: 

fn sum_cubes(a: i32, b: i32) -> i32 {
    (a..=b).map(|x| x * x * x).sum()
}

The iterator version is more concise and equally expressive. The map operation corresponds to our term parameter, the range a..=b handles iteration, and sum performs the accumulation.

The filter combinator selects elements based on a predicate, enabling elegant expression of conditional accumulation. From Exercise 1.33, we implemented filtered_accumulate to sum or multiply only elements satisfying a condition. With iterators: 

// Sum of squares of odd numbers from 1 to 10
let sum: i32 = (1..=10)
    .filter(|x| x % 2 == 1)
    .map(|x| x * x)
    .sum();

assert_eq!(sum, 165);  // 1 + 9 + 25 + 49 + 81

The filter combinator creates an iterator that yields only values for which the predicate returns true. We can chain multiple transformations, building complex pipelines from simple operations.

Lazy Evaluation and Zero-Cost Abstraction ¶

A crucial property of iterator combinators is lazy evaluation—no computation happens until a consumer demands values. When we write: 

let iter = (1..=1000000)
    .map(|x| x * x)
    .filter(|x| x % 2 == 0);

this creates an iterator pipeline but performs no computation. No array is allocated, no squares are computed, no filtering occurs. The iterator is a lightweight state machine describing how to produce values, not the values themselves.

Computation happens only when we consume the iterator: 

let first_five: Vec<i32> = iter.take(5).collect();

The take combinator creates an iterator that yields at most 5 elements, and collect consumes the iterator, producing a vector. Only then are values computed—and only the values actually needed. We compute 5 squares and perform 5 modulo operations, not a million.

This laziness enables efficient chaining. Consider: 

let sum: i32 = (1..=1000)
    .map(|x| x * 2)
    .filter(|x| x % 3 == 0)
    .map(|x| x * x)
    .sum();

A naive implementation might create three intermediate collections: one for the doubled values, one for the filtered values, and one for the squared values. Instead, the Rust compiler fuses these operations into a single loop with no allocations: 

// Equivalent to hand-written:
let mut sum = 0;
for x in 1..=1000 {
    let doubled = x * 2;
    if doubled % 3 == 0 {
        let squared = doubled * doubled;
        sum += squared;
    }
}

This is zero-cost abstraction: the iterator version compiles to the same efficient machine code as the manual loop, but expresses the computation more declaratively.

The Fold Combinator ¶

The sum and product methods are special cases of a more general pattern: folding. The fold combinator corresponds directly to our accumulate from Exercise 1.32: 

fn accumulate<T, F, C>(
    combiner: C,
    null_value: T,
    term: F,
    a: i32,
    next: fn(i32) -> i32,
    b: i32,
) -> T
where
    F: Fn(i32) -> T,
    C: Fn(T, T) -> T,
    T: Copy,
{
    if a > b {
        null_value
    } else {
        combiner(
            term(a),
            accumulate(combiner, null_value, term, next(a), next, b),
        )
    }
}

With fold: 

let result = (a..=b)
    .map(term)
    .fold(null_value, combiner);

The fold method takes an initial value and a binary function, applying the function to the accumulator and each element in turn. For example, factorial can be expressed as: 

fn factorial(n: u64) -> u64 {
    (1..=n).fold(1, |acc, x| acc * x)
}

This is an iterative process in the sense of Linear Recursion and Iteration—the accumulator holds the complete state, and the computation proceeds linearly through the range.

Rich Combinator Library ¶

Rust’s standard library provides dozens of iterator combinators, each capturing a common pattern. Here are some essential ones:

chain

Concatenates two iterators sequentially. 

let combined: Vec<i32> = (1..=3)
    .chain(10..=12)
    .collect();
assert_eq!(combined, vec![1, 2, 3, 10, 11, 12]);
zip

Pairs elements from two iterators. 

let pairs: Vec<(i32, char)> = (1..=3)
    .zip(['a', 'b', 'c'])
    .collect();
assert_eq!(pairs, vec![(1, 'a'), (2, 'b'), (3, 'c')]);
enumerate

Adds indices to iterator elements. 

let indexed: Vec<(usize, char)> = ['a', 'b', 'c']
    .iter()
    .enumerate()
    .collect();
assert_eq!(indexed, vec![(0, 'a'), (1, 'b'), (2, 'c')]);
take

Yields at most n elements. 

let first_three: Vec<i32> = (1..)  // Infinite iterator!
    .take(3)
    .collect();
assert_eq!(first_three, vec![1, 2, 3]);
skip

Skips the first n elements. 

let after_skip: Vec<i32> = (1..=10)
    .skip(5)
    .collect();
assert_eq!(after_skip, vec![6, 7, 8, 9, 10]);

These combinators compose freely. We can express complex queries concisely: 

// Find the sum of the squares of the first 5 even Fibonacci numbers
fn fib_iter() -> impl Iterator<Item = u64> {
    let mut a = 0;
    let mut b = 1;
    std::iter::from_fn(move || {
        let current = a;
        a = b;
        b = current + b;
        Some(current)
    })
}

let sum: u64 = fib_iter()
    .filter(|x| x % 2 == 0)
    .take(5)
    .map(|x| x * x)
    .sum();

assert_eq!(sum, 29480);  // 0 + 4 + 64 + 4624 + 24784

This pipeline: generates an infinite sequence of Fibonacci numbers, filters for even values, takes the first 5, squares each, and sums the results. All without allocating intermediate collections, all compiled to efficient machine code.

Collect and Type Inference ¶

The collect method deserves special attention. It consumes an iterator and builds a collection—but which collection? The target type is determined by type inference: 

let v: Vec<i32> = (1..=5).collect();
let a: [i32; 5] = (1..=5).collect();  // Array (size must match!)

When inference is ambiguous, we use the “turbofish” syntax ::<T>: 

let v = (1..=5).collect::<Vec<i32>>();

Many collection types implement FromIterator, the dual of IntoIterator. This trait describes how to build a collection from an iterator, enabling collect to work with vectors, arrays, sets, maps, and custom collections.

Comparison with Recursive Approaches ¶

Throughout Chapter 1, we’ve emphasized recursive procedures and the processes they generate. Iterators offer a complementary perspective: computation as data transformation pipelines. Consider the sum_of_squares procedure from earlier sections: 

// Recursive version
fn sum_of_squares_rec(a: i32, b: i32) -> i32 {
    if a > b {
        0
    } else {
        (a * a) + sum_of_squares_rec(a + 1, b)
    }
}

// Iterative version (tail-recursive)
fn sum_of_squares_iter(a: i32, b: i32) -> i32 {
    fn iter(a: i32, b: i32, acc: i32) -> i32 {
        if a > b {
            acc
        } else {
            iter(a + 1, b, acc + (a * a))
        }
    }
    iter(a, b, 0)
}

// Iterator version
fn sum_of_squares_range(a: i32, b: i32) -> i32 {
    (a..=b).map(|x| x * x).sum()
}

All three are equivalent in their mathematical meaning and in their compiled performance (modern Rust compilers optimize all three similarly). The choice among them is stylistic and depends on the problem structure.

Recursion shines when the problem has natural recursive structure (tree traversal, divide-and-conquer algorithms). Iteration shines for sequential processing where the transformation pipeline is clear. Rust’s iterators give us the declarative elegance of higher-order functions with the performance of low-level loops.

Infinite Iterators ¶

Like the infinite streams we’ll explore in Infinite Streams, iterators can represent unbounded sequences. The range 1.. generates all positive integers (until overflow). We can create infinite iterators that generate values lazily: 

// All positive integers
let naturals = 1..;

// All powers of 2
let powers_of_2 = std::iter::successors(Some(1u64), |n| n.checked_mul(2));

// All Fibonacci numbers (until overflow)
let fibs = std::iter::successors(Some((0u64, 1u64)), |(a, b)| {
    Some((*b, a + b))
}).map(|(a, _)| a);

These infinite iterators are safe because iteration is lazy. We must combine them with take, take_while, or similar combinators to extract finite subsequences: 

let first_10_fibs: Vec<u64> = fibs.take(10).collect();

This pattern—infinite data structures made practical by lazy evaluation—anticipates the stream processing we’ll study in Chapter 3.
